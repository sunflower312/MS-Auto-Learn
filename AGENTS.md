## 项目定位

- 本仓库是面向 VPS / Linux 部署的 Web App。
- 仓库采用 `apps/web`、`apps/api`、`workers/`、`packages/contracts` 的分层结构。
- 核心能力包括：
  - 账号池管理
  - 计划池管理
  - 运行域编排
  - Microsoft Learn 自动化流程
  - AI 辅助答题
  - 可视化运行状态、进度、错误与启停控制

## 架构约束

- 用户侧能力通过 Web App 提供，界面状态由前端应用管理。
- 浏览器自动化与 AI 答题逻辑必须保留在 API / Worker 侧，不能放到用户浏览器执行。
- 并发、重试、取消、遥测、失败恢复必须围绕明确的 `job / run` 模型设计。
- 涉及跨层能力时，先收敛类型化 contracts、DTO 与状态模型，再展开实现。
- 新增目录或模块应继续沿 `apps/`、`workers/`、`packages/` 的分层方向演进。
- 保持模块职责清晰，避免把 UI、编排、自动化、持久化重新混回少量大文件。

## 技术方向

- Web 前端固定使用 React 19、TypeScript、Vite 8、Tailwind CSS v4、Radix UI、Lucide React、React Router v7，以及 `clsx` + `tailwind-merge` + `class-variance-authority`。
- Radix primitives 应封装为项目内可复用组件，不依赖 shadcn CLI 生成作为结构前提。
- 命令与脚本默认使用 POSIX shell，并保持 Linux-first 的路径与运行假设。

## 测试与验收

- 涉及 `Microsoft Learn` 自动化链路的实现与验收时，只接受真实 E2E 作为验证方式。
- 本地确定性逻辑使用 focused tests；共享 contracts 以 typecheck 和消费方验证为准。
- 没有新鲜验证证据，不得声称完成。
- 测试入口与命令以 [docs/testing/README.md](docs/testing/README.md) 为准。

## 协作约定

- 默认使用简体中文沟通；代码标识符、类型名、函数名、组件名保持英文。
- 输出进度时应明确区分已确定结论与未解决风险。
- 如果任务结束在分析或准备阶段，必须明确说明仓库是否已准备好进入下一步实现。

### 面向用户文本

- 在发送面向用户的文本时，要按“为一个人写作”而不是“向控制台记录日志”的方式组织语言。默认假设用户看不到大多数工具调用或思考过程，只能看到你的文本输出。在第一次调用工具之前，先简短说明接下来要做什么。处理过程中，在发现关键问题、改变方向或已经取得进展但还没有更新时，给出简短更新。
- 在给出更新时，要假设对方曾经离开过，现在已经跟不上上下文。他们不知道你在过程中创造的代号、缩写或简写，也没有持续追踪处理过程。要写得让他们即使在毫无上下文的情况下也能重新接上：使用完整且语法正确的句子，不要使用未经解释的术语。技术术语要写出全称并加以说明。解释宁可多一点，也不要太少。要留意用户专业水平的线索；如果对方看起来是专家，可以稍微简洁一些；如果对方看起来是新手，就应当解释得更充分一些。
- 面向用户的文字应当使用连贯流畅的散文式风格，避免使用残句、过多的长破折号、符号、记号，或其他类似难以理解的表达。只有在确实合适时才使用表格；例如，用来承载简短且可枚举的事实，如文件名、行号、通过或失败，或者传达定量数据。不要把解释性的推理塞进表格单元格里，应当在表格之前或之后说明。避免语义回跳；要组织好每一句话，使人能够顺着线性阅读，在不必回头重新解析前文的情况下逐步建立理解。
- 最重要的是让读者在没有额外心理负担、也无需追问的情况下理解你的输出，而不是一味追求简短。如果用户必须反复重读摘要，或者还要再来问你解释，那所浪费的时间远远超过第一次阅读时少写几句所节省的时间。回应要与任务匹配：一个简单问题就直接用散文式文字回答，不要加标题和编号分节。在保持表达清晰的同时，也要保持简洁、直接、没有废话。避免填充内容，也不要陈述显而易见的事情。直接进入重点。不要过度强调与你处理过程有关但并不重要的细节，也不要用夸张措辞去渲染微小的得失。在合适的情况下使用“倒金字塔”结构，先说行动或结论；如果推理或过程中有某些内容极其重要，必须出现在面向用户的文本中，就把它放在最后。
- 这些关于面向用户文本的说明不适用于代码或工具调用。
